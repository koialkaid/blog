## 先定边界，再谈执行

让 Agent 做事之前，我现在更关心的不是它能不能做，而是它应该在什么边界里做。

这个顺序很重要。如果边界不清楚，能力越强，越容易把问题扩大。

## 哪些边界最基础

最基础的边界通常不是抽象原则，而是一些很具体的限制：

- 哪些目录可以改
- 哪些文件不能碰
- 是否允许联网
- 失败几次后必须停止
- 是否可以创建提交或推送

这些问题听起来琐碎，但它们决定了 Agent 是在帮忙，还是在制造新的不确定性。

## 任务描述也属于边界

很多时候，边界不是额外补充，而是任务本身的一部分。

例如：

```text
只修改样式文件
不要改内容
不要改部署流程
失败两次后停止
```

这样的描述会明显降低误操作的概率。

## 一个更稳的协作方式

我更喜欢把 Agent 当成一个可以配合的人，而不是一个完全自动的机器。

这意味着我需要给它足够清楚的目标，也需要给它足够明确的限制。执行过程中，它可以自主推进，但不能把边界当成可选项。

> 好的 Agent 协作，不是让它什么都做，而是让它在正确的范围里把事情做好。

## 这一阶段先记住什么

如果要继续学习 Agent，我想先把这件事放在前面：边界不是束缚能力，而是让能力变得可用。

没有边界的自动化，很容易变成另一种不可控。真正可靠的 Agent，应该知道如何执行，也应该知道什么时候停下来。
