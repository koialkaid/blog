前两篇文章分别完成了两件事：

- 第一篇先把一则真实招聘需求摆出来；
- 第二篇再从招聘要求反推一条通用的学习路线。

但通用路线还不够。因为真正落到自己身上时，学习计划必须回答两个问题：

1. 我现在到底处在什么位置；
2. 我接下来应该按什么顺序学，才能最快接近目标岗位。

这篇文章就只做这件事：**把抽象路线落成我自己的学习任务与进度规划。**

## 一、先界定我自己的起点

如果不先承认起点，后面的路线就很容易变成自我感动式计划。

我当前的状态，大概不是“完全零基础”，但也远没有到“能胜任 Agent 工程师”的程度。

### 1. 我已经接触过什么

我日常会使用 Codex app，也接触 API、号池这类东西。

这说明我对“AI 工具层”和“调用层”不是完全陌生：

- 我知道模型可以被接入使用；
- 我知道 API 是真实工作流的一部分；
- 我也已经在用 AI 参与一些实际任务，而不是只停留在看概念。

这一步的意义在于：我不是从 0 开始认识 AI。

### 2. 我现在真正缺什么

但另一方面，我也很清楚自己缺的不是“再多用几个工具”，而是下面这些更底层的理解：

- 为什么 Agent 不是简单的“聊天 + 调 API”；
- Tool Use / Function Calling 到底解决什么问题；
- 上下文、记忆、规划、多步执行之间是什么关系；
- 为什么一个看起来能跑的 Demo，离真实产品还差很远；
- 延迟、稳定性、成本、失败兜底为什么会在招聘里被反复强调。

也就是说，我现在的问题不是“完全没碰过”，而是：

> **我在用一些东西，但还没有把这些东西连成完整的结构。**

### 3. 我现在不适合的学习方式

基于这个起点，我不适合直接走两条路：

#### 不适合路线一：上来就刷框架

比如一上来就学：

- LangChain
n- LangGraph
- Coze
- 各种 workflow builder

这样做的问题不是这些东西没用，而是我现在还没有足够稳的概念骨架。框架会让我“看起来做了很多”，但不代表我真正理解了底层机制。

#### 不适合路线二：上来就做重项目

我当然可以立刻做项目，但如果理论没有补起来，项目大概率会变成：

- 能跑
- 但解释不清楚
- 换一个场景就不会做
- 面试时说不明白为什么这样设计

所以我给自己的路线定义很明确：

> **理论优先，但不是纯空谈；先建立结构化理解，再进入有针对性的实践。**

## 二、我的目标不是“学会 AI”，而是“往 Agent / AI 工程师方向转”

这个目标会直接决定学习顺序。

如果我的目标只是“把 AI 用顺一点”，那我可能只需要继续熟悉工具、优化提示词、积累一些 workflow 使用经验。

但我现在更偏向的是：

> **尽快往 Agent / AI 工程师方向转。**

这意味着我不能只学“怎么用”，而要开始补：

- 技术原理
- 系统意识
- 能力分层
- 工程语境里的抽象能力

换句话说，我不是想成为“会用 AI 的人”，而是想成为：

> **能理解 Agent 为什么这样工作，并逐步有能力把它做成产品的人。**

## 三、我为什么把学习重点放在理论上

很多人一听“尽快转向”，就会本能觉得应该疯狂做项目。

但对我来说，现在最稀缺的不是项目数量，而是**理论抓手**。

### 1. 我现在最需要的是概念地图

我现在需要先把这些词真正分清：

- LLM
- Prompt
- Context
- Memory
- Tool Use
- Function Calling
- Agent
- Workflow
- Planning
- RAG
- Multi-agent

如果这些词没有先形成一张图，那后面学什么都容易碎片化。

### 2. 我需要的是“工程语境中的理论”

我不是去读纯学术课，也不是要从 Transformer 数学推导开始。

我真正要补的是：

- 这个概念在实际系统里解决什么问题；
- 它和别的模块怎么连接；
- 为什么招聘里会看重它；
- 它的边界在哪里；
- 它出问题时通常会表现成什么样。

所以我给自己定的学习方式是：

> **理论优先，但每个理论都必须带着工程问题一起理解。**

## 四、学习主线怎么排：先学什么，后学什么

如果按我现在的基础和目标来排，我会把学习顺序定成下面这样。

### 第一阶段：先建立 Agent 的基础骨架

这部分是最优先的，因为它直接决定我后面能不能理解岗位描述里的关键词。

#### 第一组：LLM 基础机制

目标不是成为模型研究员，而是要搞清楚：

- 模型的输入输出到底是什么；
- system / user / assistant 为什么这样分；
- 上下文为什么会影响结果；
- token、窗口长度、成本、延迟之间是什么关系；
- 为什么模型并不真正“理解”，却仍然能完成很多任务。

如果这一层不清楚，后面的 Prompt、Tool Use、RAG 都会学得很飘。

#### 第二组：Prompt / 上下文 / 输出约束

我要搞清楚的不是“Prompt 技巧大全”，而是：

- Prompt 到底在控制什么；
- 为什么有时约束写了但模型不听；
- 为什么多轮对话会越来越乱；
- 如何让输出更稳定；
- 什么时候应该让模型自由回答，什么时候应该强结构化。

#### 第三组：Function Calling / Tool Use

这是我必须尽快补齐的一块。

因为从招聘来看，这份岗位核心不是“回答得像个人”，而是：

- 能不能调用工具；
- 能不能基于工具结果继续执行；
- 能不能把用户问题变成一串可执行动作。

我对这一块目前是“知道有，但没真正系统懂”。所以这是第一优先级里的重点。

### 第二阶段：理解 Agent 为什么能多步执行

当基础骨架有了以后，我要进入第二层：

#### 1. 任务拆解

- 一个复杂任务为什么要拆成多个步骤；
- 拆得太粗和拆得太细分别会出什么问题；
- 哪些步骤该交给模型，哪些步骤该交给规则或工具。

#### 2. 工作流与规划

- 什么叫 planning；
- 什么只是顺序执行，什么才算真正的规划；
- 一个 Agent 的“思考”本质上是不是状态流转。

#### 3. 多步执行中的状态

- 当前步骤是什么；
- 上一步结果如何影响下一步；
- 中间状态应该保存什么；
- 失败时怎么继续而不是从零重来。

这一层是我从“会调用 LLM”转向“开始理解 Agent 系统”的关键。

### 第三阶段：再补更热、更容易被误学的词

这时候再去碰下面这些，才比较稳：

#### 1. RAG

我现在工作里其实还没有明显的知识库 / 检索增强场景，所以 RAG 不是第一优先级。

但它是招聘里的高频加分项，所以后面必须补。

只是顺序上要放在 Agent 核心机制之后。

#### 2. LangChain / LangGraph

我现在看到这些词很多次，但我不准备把它们当成起点。

原因很简单：

- 先学框架，容易学成“会拼积木”；
- 后面一旦脱离框架，就不知道系统为什么这样跑。

所以我对它们的定位是：

> 等底层结构稍微稳了之后，再把它们当作加速器。 

## 五、我的学习任务应该怎么拆

我现在每天可投入时间不稳定，但平均有 **2~4 小时**。

这个强度不算轻，所以完全可以走一条“高密度但不失控”的路线。

我打算把学习任务拆成两类：

### 1. 主任务：理论理解

这是每天都要推进的主线。

每次学习都围绕一个核心问题展开，比如：

- LLM 输出为什么会漂？
- 上下文到底怎么影响模型？
- Function Calling 和普通对话的本质区别是什么？
- Agent 为什么不是“多问模型几次”那么简单？
- 为什么真实产品里稳定性和成本会压过 Demo 效果？

也就是说，我不按“看章节”学，而按“解问题”学。

### 2. 副任务：最小验证

虽然我说理论优先，但每学完一块，我还是要做一点点最小验证。

不是做完整项目，而是做能验证理解的小实验，比如：

- 用最小脚本观察不同 Prompt 对结构化输出的影响；
- 看一次 function calling 的完整请求与返回；
- 把一个任务手工拆成多步，再和 Agent 框架的方式对照；
- 观察一次多轮上下文过长导致回答变差的现象。

这样做的目的不是“立刻做作品集”，而是防止理论变成空记忆。

## 六、我的进度规划：按 6 周压缩推进

我现在是“越快越好”的状态，但又不适合直接暴力做项目，所以我会给自己定一个 **6 周压缩版** 计划。

## 第 1 周：把概念地图搭起来

### 目标

先把高频词的关系理顺，不再看到一个词就慌一次。

### 重点任务

- LLM 是什么，不是什么；
- Prompt / Context / Token / Window 的关系；
- Function Calling / Tool Use 的区别与联系；
- Agent、Workflow、Planning 的关系；
- RAG、Memory、Multi-agent 各自放在哪个层级。

### 本周产出

- 一份自己的术语关系图；
- 每个核心概念写出一句“工程语境下的定义”；
- 明确哪些词是现在优先学，哪些词先压后。

## 第 2 周：补 LLM 调用与上下文机制

### 目标

把“我会接 API”升级成“我知道它为什么这么工作”。

### 重点任务

- 理解消息结构；
- 理解 system prompt 的作用边界；
- 理解上下文污染与上下文裁剪；
- 理解结构化输出为什么容易出问题。

### 本周产出

- 一份“LLM 调用最常见问题清单”；
- 几个最小实验记录；
- 一篇自己的总结：为什么调用模型不等于理解模型。

## 第 3 周：集中补 Function Calling / Tool Use

### 目标

把这块从“知道有”提升到“能说清楚为什么重要”。

### 重点任务

- 什么情况下模型应该调用工具；
- 为什么函数参数要结构化；
- 工具返回结果如何回流给模型；
- 工具失败时为什么会让 Agent 整体变脆。

### 本周产出

- 一份 Tool Use 流程图；
- 一份 Function Calling 的关键问题笔记；
- 最小脚本实验：一次完整调用链路。

## 第 4 周：学习 Agent 工作流与多步执行

### 目标

真正理解“一个 Agent 为什么能持续工作”。

### 重点任务

- 任务拆解
- 步骤状态
- 中间结果保存
- 停止条件
- 错误恢复

### 本周产出

- 一张多步执行流程图；
- 一篇笔记：Agent 和普通对话系统的差别；
- 一个最小工作流原型或伪代码设计。

## 第 5 周：补系统设计视角

### 目标

让自己开始从“功能”转到“系统”。

### 重点任务

- 延迟为什么重要；
- 成本为什么会影响方案设计；
- 为什么要做失败兜底；
- 为什么可控性比 Demo 效果更重要；
- “能力 ≠ 体验”在产品里意味着什么。

### 本周产出

- 一篇“如果我要做一个可上线 Agent，我最怕什么”总结；
- 一份系统风险清单。

## 第 6 周：再进入 RAG / 框架

### 目标

这时才开始碰那些高频热词。

### 重点任务

- RAG 解决什么问题；
- LangChain / LangGraph 在系统里解决什么问题；
- 为什么它们是加速器，而不是理解起点。

### 本周产出

- 一份“我现在适合学哪些框架、不适合学哪些”的判断；
- 一份下一阶段的实践计划草案。

## 七、每天 2~4 小时，怎么安排更现实

因为我的时间不稳定，所以我不能用那种特别理想化的作息表。

我更适合的是三段式：

### 模式 A：2 小时版本

- 1 小时：理论阅读 / 概念整理
- 30 分钟：笔记归纳
- 30 分钟：最小验证 / 回顾

### 模式 B：4 小时版本

- 1.5 小时：理论主任务
- 1 小时：对照示例 / 官方文档 / 关键资料
- 1 小时：最小实验或结构图整理
- 30 分钟：写结论与问题清单

### 每天最低要求

即使很忙，我也希望自己做到：

- 至少推进一个概念；
- 至少写下一条清晰结论；
- 不让学习只停在“今天看了很多”。

## 八、什么算我真的在进步

对我来说，进步不应该用“今天看了几个教程”来算。

我更应该用下面这些标准判断：

### 1. 我能不能解释清楚

比如：

- 什么是 Function Calling；
- 为什么 Agent 需要状态；
- 为什么 RAG 不是万能增强；
- 为什么多 Agent 不是越早学越好。

### 2. 我能不能区分边界

比如：

- 哪些是模型问题；
- 哪些是系统问题；
- 哪些是产品问题；
- 哪些是工程实现问题。

### 3. 我能不能把一个词放回系统位置里

这才是最重要的。

如果我看到一个词，不再只是“我见过”，而是知道：

- 它解决什么问题；
- 它和什么配合；
- 它什么时候该用；
- 它为什么会失败；

那我才算真正开始往 Agent / AI 工程师方向走了。

## 九、我暂时不做什么

为了保证路线不跑偏，我也要明确写下来：

### 暂时不优先做的事

- 不把 LangChain 当起点；
- 不一上来沉迷 multi-agent；
- 不为了“看起来厉害”去堆概念；
- 不把做项目当成掩盖理解不足的手段；
- 不因为焦虑而同时开 10 条学习线。

### 暂时不追求的结果

- 不是马上做出最炫的 Demo；
- 不是一下子补齐所有工程能力；
- 不是一周变成岗位匹配度很高的人。

我现在更务实的目标是：

> **先把理论骨架搭起来，再让后面的每一步都踩在结构上。**

## 十、小结：这是一条“先搭骨架，再长肌肉”的路线

如果把这份规划压缩成一句话，它其实就是：

> 先从“会用 AI 工具的人”，变成“理解 Agent 系统是怎么工作的那种人”，再继续向 Agent / AI 工程师靠近。

这条路看起来没有直接做项目那么刺激，但它更适合我现在的阶段。

因为我现在真正稀缺的，不是“手又做了一个东西”，而是：

- 一套稳定的概念骨架；
- 一条不被热词带跑的学习顺序；
- 一种能持续积累的节奏感；
- 一种从使用者走向工程视角的转变。

如果我能先把这些补起来，后面再补工程、框架、项目，才会真正越走越稳。
